1~4줄의 간단한 명령·프롬프트를 받아 `.claude/specs/{slug}.md` 상세 제품 스펙 파일을 생성한다.

## /task와의 차이

| | /task | /spec |
|---|---|---|
| 목적 | 구현 지침 (파일·라인 기준) | 제품 요구사항 정의 |
| 출력 | 코드 변경 체크리스트 | 기능 명세·완료 기준 |
| 시점 | 구현 직전 | 구현 전 설계 단계 |

## 절차

1. 입력 문장에서 기능의 핵심 의도와 범위를 파악한다
2. 관련 기존 코드·컴포넌트를 **직접 읽어서** 재사용 가능 항목과 영향 범위를 확인한다 — 추측 금지
3. 아래 형식으로 스펙 파일을 작성한다
4. 파일명은 기능을 나타내는 kebab-case로 한다 (예: `user-point-monthly-view.md`)

## 스펙 파일 형식

```markdown
# 스펙: {기능 제목}

## 개요
기능의 목적과 배경 (2~3문장)

---

## 사용자 시나리오
누가, 어떤 상황에서, 무엇을 할 수 있는지 (불릿 또는 번호 목록)

---

## 기능 요구사항
- [ ] 필수 기능 항목 (반드시 구현)
- [ ] 선택 기능 항목 (Nice to have)

---

## UI/UX 명세

### 화면 구성
페이지·섹션별 레이아웃 설명

### 상태 정의
| 상태 | 표시 내용 | 동작 |
|---|---|---|
| 로딩 | ... | ... |
| 정상 | ... | ... |
| 오류 | ... | ... |
| 빈 상태 | ... | ... |

---

## 기술 스펙

### 재사용 가능한 기존 패턴
- `경로/컴포넌트명` — 재사용 방식

### 신규 개발 필요 항목
- 새로 만들어야 할 컴포넌트·함수

### 영향 범위
- 수정이 예상되는 파일 목록

---

## 제약 조건 / 고려사항
- 기존 코드와의 호환성
- 반응형 대응 범위 (모바일 / 태블릿 / 데스크탑)
- 접근성, 성능 등

---

## 완료 기준 (Definition of Done)
- [ ] 기능 동작 검증 항목
- [ ] `pnpm build` 통과
- [ ] `pnpm lint` 통과
- [ ] 반응형 확인 항목
```

## 주의

- 파일을 읽지 않고 "재사용 가능한 기존 패턴" 섹션을 채우지 않는다
- 영향 범위는 실제 코드 탐색 결과로만 기술한다
- 입력이 모호한 경우 먼저 범위를 질문한다 (추측하여 작성하지 않는다)
- 이 커맨드는 구현 지침이 아닌 **무엇을 만들지**를 정의하는 문서다
