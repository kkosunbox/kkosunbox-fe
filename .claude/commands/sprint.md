1~4줄의 스프린트 목표를 받아 `.claude/sprints/{slug}-contract.md` 계약 파일을 생성한다.
생성기(Generator)와 평가기(Evaluator)가 "완료"의 기준을 사전에 합의하기 위한 파일이다.

## /task·/spec과의 차이

| | /spec | /task | /sprint |
|---|---|---|---|
| 목적 | 제품 요구사항 정의 | 구현 지침 (파일·라인) | 완료 기준 합의 + 평가 기준 설정 |
| 소비자 | 개발자 | 개발자 | 평가기(/evaluate) |
| 시점 | 설계 | 구현 직전 | 스프린트 시작 전 |

## 절차

1. 입력에서 스프린트 목표와 범위를 파악한다
2. 관련 기존 코드·컴포넌트를 **직접 읽어서** 구현 범위와 재사용 가능 항목을 확인한다
3. 완료 기준을 Playwright로 **테스트 가능한 형태**로 구체화한다
4. 평가 기준별 최저 통과 점수를 제안한다
5. 아래 형식으로 계약 파일을 작성한다 — 상태는 항상 `협의중`으로 시작
6. 파일명은 기능을 나타내는 kebab-case slug로 한다 (예: `point-monthly-view`)

## 계약 파일 형식 (`.claude/sprints/{slug}-contract.md`)

```markdown
# 스프린트 계약: {제목}

> 상태: 협의중  
> 생성일: {날짜}  
> slug: {slug}

---

## 목표
이번 스프린트에서 달성할 핵심 목표 (1~2문장)

---

## 구현 범위
- [ ] 구현할 항목 1
- [ ] 구현할 항목 2

## 구현 제외 (이번 스프린트 범위 밖)
- 제외 항목 (이유 포함)

---

## 완료 기준 (Playwright 테스트 가능 형태)
각 항목은 `/evaluate` 실행 시 실제로 검증되어야 한다.

- [ ] `http://localhost:3000/{경로}` 접속 시 {요소}가 렌더링된다
- [ ] {버튼/탭}을 클릭하면 {결과}가 표시된다
- [ ] {상태} 에서 {요소}는 {동작}한다
- [ ] `pnpm build` 오류 없이 통과
- [ ] `pnpm lint` 오류 없이 통과

---

## 평가 기준 및 기준치

| 기준 | 최저 통과 | 검증 방법 |
|---|---|---|
| 기능성 | 완료 기준 80% 이상 통과 | Playwright UI 테스트 |
| 제품 완성도 | 미완성 항목 없음 | 구현 범위 체크리스트 |
| 시각 디자인 | CLAUDE.md 디자인 규칙 위반 없음 | /design-check |
| 코드 품질 | build + lint 통과 | pnpm build && pnpm lint |

하나라도 미달 시 스프린트 **실패** 처리.
```

## 주의

- 완료 기준은 반드시 Playwright로 자동 검증 가능한 형태로 작성한다 ("잘 보인다" 같은 주관적 표현 금지)
- 코드를 읽지 않고 구현 범위를 추측하지 않는다
- 이 파일은 `/evaluate` 커맨드가 읽는 입력이다 — 형식을 임의로 바꾸지 않는다
- 상태가 `승인됨`이 되기 전에는 구현을 시작하지 않는 것을 권장한다
