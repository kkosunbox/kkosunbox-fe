현재 작업 중인(스테이징/언스테이징/untracked 포함) 변경 사항을 **Playwright E2E 테스트로 검증한 뒤, task 단위로 나눠 여러 개의 커밋**으로 만든다.

인자: (선택) 커밋 범위 힌트나 task 분할 지시. 없으면 `git status`의 모든 변경을 대상으로 자동 분할한다.

## 핵심 정책

1. **테스트 실패 시 중단한다.** Playwright 테스트가 하나라도 실패하면 **커밋하지 않고** 실패 내용을 보고한 뒤 멈춘다. 사용자가 명시적으로 무시를 지시하지 않는 한 진행하지 않는다.
2. **요청 범위만 커밋한다.** 사용자가 만들지 않은/설명과 다른 변경을 발견하면 멈추고 보고한다 (CLAUDE.md 작업 범위 준수 규칙).
3. **커밋·푸시는 이 명령에서만 수행한다.** 푸시는 하지 않는다(사용자가 따로 지시할 때만).

## 절차

### 1단계 — 변경 파악 및 task 분할 계획
- `git status`와 `git diff`(untracked는 내용 확인)로 전체 변경을 읽는다.
- 변경을 **논리적 task 단위**로 묶는다. 묶는 기준:
  - 같은 기능/화면/관심사(예: order 액션 칩 통일, hero 이미지 사이즈, package-plans 동기화)
  - `.claude/` 메타 파일(settings, commands, 스크린샷 등)은 별도 `chore` 커밋으로 분리
  - 함께 동작해야 의미가 있는 파일(예: 컴포넌트 + 그 컴포넌트가 import하는 신규 훅)은 같은 커밋
- 분할 계획(커밋 N개, 각 커밋의 파일·메시지 초안)을 **먼저 출력**한다.

### 2단계 — 정적 검증 (CLAUDE.md 작업 검증 규칙)
```
pnpm tsc --noEmit   # 또는 npx tsc --noEmit
pnpm lint
```
- **신규** 오류만 수정 대상. 기존부터 있던 오류는 건드리지 않는다.
- 신규 타입/린트 오류가 있으면 고치고 다시 검증한다. 못 고치면 중단·보고.

### 3단계 — Playwright E2E 테스트
```
pnpm test
```
- 최초 실행은 `next build`(~2분)를 포함해 최대 5분 소요. port 3001 서버가 살아있으면 빌드 없이 재사용된다.
- **코드를 방금 바꿨으므로, 빌드 캐시가 오래됐을 수 있다.** 이전 빌드를 재사용한 결과처럼 보이면(변경이 반영 안 된 통과) port 3001 프로세스를 종료하고 다시 돌린다.
- 결과 판정:
  - **전부 통과** → 4단계로 진행
  - **하나라도 실패** → **여기서 중단.** 실패한 spec·테스트명·오류 요약을 보고하고 커밋하지 않는다. (핵심 정책 1)

### 4단계 — task별 커밋
- 1단계 계획대로 `git add <해당 파일들>` → `git commit` 을 task마다 반복한다.
- 커밋 메시지는 이 저장소 컨벤션(한국어, `feat:`/`fix:`/`chore:` 프리픽스, 무엇을 왜 바꿨는지)을 따른다. 최근 `git log`로 톤을 맞춘다.
- 현재 브랜치가 기본 브랜치(main)면 먼저 작업 브랜치를 만든다.

### 5단계 — 결과 보고
- 만든 커밋 목록(`git log --oneline -N`)
- tsc / lint / playwright 결과 한 줄 요약
- 커밋하지 않고 남겨둔 변경이 있으면 그 이유

## 출력 형식

- 분할 계획 표 (커밋 # · 파일 · 메시지)
- 정적 검증 결과 (tsc / lint)
- Playwright 결과 (통과/실패, 실패 시 상세)
- 최종 커밋 로그
