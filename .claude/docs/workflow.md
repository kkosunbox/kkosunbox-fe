# AI 협업 개발 워크플로우

> 이 문서는 Claude Code와 협업하는 방식의 철학과 커맨드 설계 의도를 설명한다.

---

## 왜 이런 구조를 만들었나

AI가 작성한 코드는 겉으로는 완성도 있어 보이지만, 실제로 실행해 보면 버그가 있는 경우가 많다.
이를 방지하기 위해 두 가지 원칙을 도입했다.

1. **구현 전에 "완료"의 기준을 명문화한다** — 모호한 스펙으로 구현을 시작하면 완성도 판단이 주관적이 된다.
2. **평가는 실제 실행으로 증명한다** — 코드 리뷰만으로는 런타임 버그를 잡지 못한다. Playwright로 브라우저를 직접 조작해 검증한다.

---

## 커맨드 지도

```
[아이디어 / 요구사항]
        │
        ▼
    /spec             → 무엇을 만들지 정의 (제품 요구사항)
        │
        ▼
    /sprint           → 이번 단위의 완료 기준 합의 (스프린트 계약)
        │
        ▼
    /task             → 어떻게 구현할지 계획 (파일·라인 기준)
        │
        ▼
    [구현]
        │
        ▼
    /evaluate         → 실제 앱을 실행해 계약 기준으로 평가
        │
      ┌─┴─────────────────┐
   통과 ✅             실패 ❌
      │                   │
   다음 /sprint    --next-sprint 로 재도전 계약 초안
```

---

## 각 커맨드의 역할

### `/spec` — 제품 명세

**언제:** 기능의 범위와 요구사항이 머릿속에만 있을 때

- 1~4줄의 짧은 설명을 상세 제품 스펙으로 확장
- 사용자 시나리오, 기능 요구사항, 완료 기준, 기존 코드베이스의 재사용 가능 항목을 함께 정의
- 출력: `.claude/specs/{slug}.md`

**`/task`와의 차이:** `/spec`은 "무엇을"이고, `/task`는 "어떻게"다.
스펙이 확정되기 전에 `/task`를 쓰면 잘못된 것을 정확하게 만드는 상황이 생긴다.

---

### `/sprint` — 스프린트 계약

**언제:** 구현을 시작하기 직전, 이번 단위의 완료 기준을 명문화할 때

- 완료 기준을 Playwright로 **자동 검증 가능한 형태**로 구체화
- 평가 기준별 최저 통과 점수를 사전에 확정
- 상태가 `승인됨`이 되기 전에는 구현을 시작하지 않는다
- 출력: `.claude/sprints/{slug}-contract.md`

**핵심 설계 의도:** 구현 완료 후에 "이게 맞나?" 논쟁이 생기지 않도록, 시작 전에 성공 기준에 합의한다.
계약은 파일로 저장되기 때문에 세션이 바뀌어도 맥락이 유지된다.

---

### `/task` — 구현 지침

**언제:** 계약이 승인된 후, 실제 코드 변경 계획을 세울 때

- 기존 코드를 직접 읽어 변경할 파일·함수·라인 번호를 정확히 명시
- 추측으로 작성하지 않는다
- 출력: `.claude/tasks/{slug}.md`

---

### `/evaluate` — 스프린트 평가

**언제:** 구현 완료 후, 계약 기준에 따라 실제 앱을 평가할 때

- 계약 파일의 완료 기준을 Playwright로 항목별 검증
- 기능성 / 제품 완성도 / 시각 디자인 / 코드 품질 네 기준으로 채점
- 하나라도 기준치 미달 시 스프린트 **실패** — 상세 피드백과 함께 재도전
- 출력: `.claude/sprints/{slug}-eval.md`

**옵션:**
```
/evaluate {slug}                    # 평가 결과만
/evaluate {slug} --next-sprint      # + 다음 계약 초안 자동 생성
/evaluate {slug} --update-tasks     # + task 우선순위 재정렬
```

---

## 파일 기반 커뮤니케이션

모든 산출물은 파일로 저장된다. 세션이 달라져도 이전 세션의 맥락(계약, 평가 결과)을 그대로 이어받을 수 있다.

```
.claude/
├── specs/          # /spec 출력 — 제품 요구사항 정의
├── sprints/        # /sprint, /evaluate 출력 — 계약 + 평가 결과
└── tasks/          # /task 출력 — 구현 지침
```

---

## 언제 어떤 커맨드를 쓰나

| 상황 | 커맨드 |
|---|---|
| "이런 기능을 만들고 싶은데 스펙이 없다" | `/spec` |
| "스펙은 있는데 이번 단위를 어디까지 할지 정해야 한다" | `/sprint` |
| "계약은 됐고 어느 파일을 어떻게 바꿀지 계획이 필요하다" | `/task` |
| "구현했는데 제대로 됐는지 검증이 필요하다" | `/evaluate` |
| "현재 변경 코드의 품질을 확인하고 싶다" | `/review` 또는 `/design-check` |

---

## 축약 경로 (간단한 작업)

기능이 명확하고 범위가 작은 경우, 전체 흐름을 생략할 수 있다.

```
작은 버그 수정: /task → 구현
단순 UI 조정: /task → 구현 → /design-check
```

`/spec`과 `/sprint`는 범위가 크거나 여러 세션에 걸쳐 진행되는 작업에서 가치가 극대화된다.
