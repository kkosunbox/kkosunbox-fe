## ⚠️ 핵심 발견 — "526 KiB unused JS"는 dev 모드 전용 아티팩트였음
코드 수정(next/dynamic 전환 + Link prefetch=false) 적용 전/후를 **각각 프로덕션 빌드로 직접 대조**한 결과:
- 수정 전 코드 · 프로덕션 빌드: Perf 100, LCP 0.6s, unused JS 28 KiB
- 수정 후 코드 · 프로덕션 빌드: Perf 97, LCP 1.3s, unused JS 46 KiB (측정 노이즈 범위 내 — 두 값 모두 우수)

즉 dev 모드에서 관측된 526 KiB unused JS·TBT 220ms·Perf 75는 **Turbopack dev 번들러가 같은 디렉터리의 클라이언트 컴포넌트를 라우트 구분 없이 하나의 공유 청크(`widgets_ebca89f4._.js`)로 뭉치는 dev-only 특성** 때문이었다. Next.js App Router는 프로덕션 빌드에서 라우트별 자동 코드 스플리팅을 이미 수행하므로, `next/dynamic` 전환 여부와 무관하게 프로덕션에서는 문제가 사실상 없었다.

## 코드 변경 사항 (유지함 — 정당성)
- 8개 mypage 서브라우트(subscription·payment·password·point·review/write·withdraw·subscription/change·subscription/detail) 정적 import → `next/dynamic()` 전환
- 대시보드 카드의 서브라우트 이동 `<Link>` 4곳(SubscriptionCard 3곳 + ProfileSection 1곳)에 `prefetch={false}` 추가
- 프로덕션 스코어를 극적으로 개선하진 않았지만(원인이 dev-only였으므로), 대시보드에서 잘 안 쓰는 서브페이지 JS를 뷰포트 진입만으로 선제 prefetch하지 않게 되어 실사용 대역폭 낭비를 줄임. 백로그 액션 항목("마이페이지 전용 위젯 dynamic import")과 부합하는 정석적인 코드 스플리팅 패턴이라 유지.
- tsc/eslint 통과 확인.
